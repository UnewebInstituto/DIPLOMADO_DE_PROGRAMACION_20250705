El patrón que observas (saltos de 2 en 2: 14, 16, 18...) no es un error de la base de datos, sino una **doble petición de incremento** generada por la lógica de tu código PHP en combinación con el comportamiento automático de PostgreSQL.

Aquí te explico exactamente qué está pasando y cómo corregirlo profesionalmente.

### El Diagnóstico: El Doble Salto

El problema ocurre porque estás llamando al incremento de la secuencia **dos veces** por cada inserción:

1. **Primer Incremento (Manual):** En tu código ejecutas `$sql_archivo = "SELECT nextval('...')";`. Esto le dice a PostgreSQL: *"Dame el siguiente número y aumenta el contador"*. Si la secuencia estaba en 13, ahora pasa a **14**.
2. **Segundo Incremento (Automático):** Al ejecutar el `INSERT INTO productos(...)`, no especificas el campo `id_producto`. PostgreSQL, al ver que el campo es (presumiblemente) de tipo `SERIAL` o tiene un `DEFAULT nextval()`, dispara **automáticamente** la secuencia otra vez. Entonces, el registro se guarda con el ID **15**.
3. **La Inconsistencia:** Tu variable `$nombre_archivo` usa el valor que obtuviste manualmente (`14`), pero en la tabla el registro quedó guardado con el `15`. En tu salida de consola se ven pares porque probablemente en algún punto el código o la secuencia se desfasaron, pero el origen es esta duplicidad.

---

### La Solución Recomendada

Para mantener la integridad y que el nombre del archivo coincida exactamente con el ID de la base de datos, debes realizar el insert y recuperar el ID generado en una sola operación usando la cláusula `RETURNING`.

#### Código Optimizado:

```php
case '3':
    $detalle_imagen = getimagesize($_FILES['imagen']['tmp_name']);
    
    if ($detalle_imagen[2] == 2 || $detalle_imagen[2] == 3) {
        $extension = ($detalle_imagen[2] == 2) ? ".jpg" : ".png";

        // 1. Iniciamos una transacción para asegurar consistencia
        pg_query($enlace, "BEGIN");

        // 2. Insertamos el producto SIN el nombre del archivo aún, recuperando el ID generado por Postgres
        // Usamos RETURNING id_producto para obtener el valor real de la secuencia
        $sql = "INSERT INTO productos (nombre_producto, descripcion, precio, existencia) 
                VALUES ('$_REQUEST[nombre]', '$_REQUEST[descripcion]', '$_REQUEST[precio]', '$_REQUEST[cantidad]') 
                RETURNING id_producto";
        
        $resultado = pg_query($enlace, $sql);
        
        if ($resultado) {
            $fila = pg_fetch_result($resultado, 0, 0);
            $id_generado = $fila;

            // 3. Ahora que tenemos el ID real, definimos el nombre del archivo
            $nombre_archivo = './img/' . $id_generado . $extension;

            // 4. Actualizamos el registro con el nombre del archivo final
            $sql_update = "UPDATE productos SET nombre_archivo = '$nombre_archivo' WHERE id_producto = $id_generado";
            pg_query($enlace, $sql_update);

            // 5. Movemos el archivo físico
            if (move_uploaded_file($_FILES['imagen']['tmp_name'], $nombre_archivo)) {
                pg_query($enlace, "COMMIT");
                $mensaje = "Producto cargado con éxito. ID: $id_generado";
                $severidad = 1;
            } else {
                pg_query($enlace, "ROLLBACK");
                $mensaje = "Error al mover el archivo físico.";
                $severidad = 2;
            }
        } else {
            pg_query($enlace, "ROLLBACK");
            $mensaje = "Error al insertar en la base de datos.";
            $severidad = 2;
        }

        header('location:/menu.php?mensaje='.urlencode($mensaje).'&severidad='.$severidad);
    } else {
        $mensaje = "Formato de imagen no válido.";
        $severidad = 2;
        header('location:/menu.php?mensaje='.urlencode($mensaje).'&severidad='.$severidad);
    }
    break;

```

---

### ¿Por qué esta solución es mejor?

1. **Uso de `RETURNING**`: Es la forma más eficiente en PostgreSQL de obtener el ID generado sin llamar a la secuencia por separado. Evita los saltos de "2 en 2".
2. **Transacciones (`BEGIN/COMMIT`)**: Si por alguna razón el archivo no se puede mover a la carpeta `./img/`, el registro no se queda "huérfano" en la base de datos; se deshace la operación.
3. **Sincronización Total**: El nombre del archivo en disco siempre será idéntico al ID en la tabla, eliminando el `$data_archivo[0]++` que es una práctica riesgosa (si dos personas insertan al mismo tiempo, los IDs podrían cruzarse).

**Nota de seguridad:** Como experto, te sugiero en el futuro utilizar **Sentencias Preparadas** (`pg_prepare` y `pg_execute`) en lugar de pasar las variables `$_REQUEST` directo al SQL, para proteger Interaxia contra ataques de Inyección SQL.
